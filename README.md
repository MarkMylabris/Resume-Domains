# Domain Scorer - скоринг кандидатов по бизнес-доменам

Локальный прототип семантического ранжирования резюме относительно вакансии по **бизнес-доменам** (отрасли и типы продуктов/услуг, а не ключевые слова вакансии). Принимает свободный текст вакансии и/или готовые списки доменов. Основной расчёт выполняется **без сети** после подготовки модели и артефактов.

---

## 1. Назначение и границы прототипа

**Вход:**

- текст вакансии (файл `.txt` или встроенные сценарии);
- пул кандидатов: либо предрасчитанный JSON с доменами на резюме, либо ваши резюме в `.txt` (домены извлекаются на лету).

**Выход:**

- упорядоченный список кандидатов;
- нормализованная оценка `score` в диапазоне [0, 1];
- доля покрытия запросных доменов `hit_rate`;
- пояснение по каждому запросному домену (`details`, шаблон `explain_text`).

**Не входит в прототип:** веб-интерфейс, полноценная ATS, хранение пользователей, интеграция с HRIS без вашей обвязки.

## 2. Архитектура системы

### 2.1 Общая схема

```
                    ┌─────────────────────────────────────┐
                    │  Текст вакансии (или список доменов) │
                    └─────────────────┬───────────────────┘
                                      │
                    ┌─────────────────▼───────────────────┐
                    │  Извлечение доменов (MMR)           │
                    │  live_demo / parse_hh_resumes       │
                    │  эмбеддинг текста → словарь доменов │
                    └─────────────────┬───────────────────┘
                                      │
                    ┌─────────────────▼───────────────────┐
                    │  Веса важности (эвристика + эмбеддинги)│
                    │  estimate_domain_weights             │
                    └─────────────────┬───────────────────┘
                                      │
   Кандидат: список доменов ◄────────┼────────► scorer.rank()
                    │                │                │
                    │                │    cosine sim + co-occur + taxonomy
                    │                │    штраф покрытия, нормализация [0,1]
                    └────────────────▼───────────────────┘
                                     │
                    ┌────────────────▼───────────────────┐
                    │  Топ-N, explain_text, JSON (опц.)   │
                    └─────────────────────────────────────┘
```

### 2.2 Режим v7-hybrid (скоринг)

| Поток | Что кодируется | Как |
|--------|----------------|-----|
| **Запрос** (домены вакансии) | Текст после ручного словаря `EXPANSIONS` в `scorer.py` | Стабильные формулировки, меньше дрейфа смысла |
| **Кандидат** | Метки доменов из словаря | Эмбеддинги из `domain_embeddings_v7.npz`, построены по `expansions.json` + ручные дополнения |

Для каждой пары «запросный домен - множество доменов кандидата» берётся **лучший** косинус среди доменов кандидата. Ниже порога `SIM_THRESHOLD` (0.72) вклад смягчается (умножение на 0.30). Добавляются бонусы co-occurrence и таксономии L1/L2, затем штраф неполного покрытия и нормализация итогового `score`.

### 2.3 Основные артефакты

| Файл | Назначение |
|------|------------|
| `candidates_normalized.json` | 117 тестовых кандидатов с нормализованными списками доменов |
| `domain_embeddings_v7.npz` | Матрица эмбеддингов всех доменов словаря (порядок строк = порядок доменов) |
| `expansions.json` | Текстовые расширения меток для обогащения смысла при кодировании кандидатских доменов |
| `taxonomy.json` | Иерархия L1/L2 для таксономических бонусов |
| `cooccur_matrix.json` | Совместная встречаемость доменов (в т.ч. пересборка по HH-датасету) |
| `config.json` | Настройки LLM (только для скриптов расширений/таксономии, не для рантайм-скоринга) |

### 2.4 Модули и ответственность

| Модуль | Роль |
|--------|------|
| `scorer.py` | Ядро: `rank`, `explain`, `explain_text`, фильтр шумовых запросных доменов, CLI |
| `live_demo.py` | Извлечение доменов из текста вакансии (MMR), веса секций, демо-сценарии |
| `score_file.py` | Скоринг из файлов: вакансия `.txt` + резюме `.txt` или готовый пул |
| `parse_hh_resumes.py` | Очистка текста HH, извлечение доменов, блоклист широких доменов, экспорт JSON |
| `build_data.py` | Парсинг SQL, нормализация, эмбеддинги, опционально co-occurrence из HH |
| `build_expansions.py` | Пополнение `expansions.json` (Wikipedia, Wikidata, LLM через `llm_client.py`) |
| `llm_client.py` | Единый вызов Gemini / OpenAI-совместимых API с повтором при лимитах |
| `demo_hh.py` | Демо на большом пуле HH (`candidates_hh_full.json`) |
| `demo.py` | Демо на 117 кандидатах по доменам |
| `test_scenarios.py` | Автоматические сценарии приёмки (23 теста) |

## 3. Стек технологий

| Компонент | Инструмент |
|-----------|------------|
| Язык | Python 3.10+ |
| Эмбеддинги | `sentence-transformers`, модель `paraphrase-multilingual-MiniLM-L12-v2` (384 измерения) |

| Зависимости | каталог `.venv_libs/` в проекте |

## 5. Конвейер выполнения на реальном примере (пошагово)

Ниже показан полный путь для сценария «вакансия QA + пул резюме HH», как в `demo_hh.py`. Это тот же принцип, что и для `score_file.py`, если вместо HH указать свои `.txt` резюме.

### Шаг 0. Исходные данные

**Текст вакансии** (фрагмент, свободная форма):

```
Нам нужен специалист по автоматизированному тестированию.
Обязательные требования:
  • Опыт автоматизации тестирования от 3 лет (Selenium, Appium, pytest)
  • Глубокое понимание QA-процессов и обеспечения качества ПО
  • Навыки API-тестирования (Postman, REST Assured)
  • SQL для верификации данных
Желательно: опыт в финтех или e-commerce проектах, знание CI/CD пайплайнов.
```

**Резюме из датасета HH** (пример сырой строки до очистки):

```
Опыт работы 11 лет 4 месяца  Январь 2018 - по настоящее время 1 год 5 месяцев
ООО "ТеТа" Воронеж  Информационные технологии, системная интеграция, интернет ...
Системный администратор  Обслуживание и поддержка IT-инфраструктуры объектов
общепита. Работа с ПО iiko. Настройка, обслуживание ККТ и POS-терминалов.
Занятость: полная занятость  График работы: полный день
```

Кандидаты в рантайме демо HH берутся из `datasets/hh_ru_resumes/candidates_hh_full.json` (после парсинга и балансировки по должностям).

### Шаг 1. Очистка текста резюме (`clean_resume` в `parse_hh_resumes.py`)

Удаляются шаблоны, мешающие семантике: периоды работы, зарплаты, «Занятость», «График», юрформы, лишняя пунктуация. После очистки тот же фрагмент ближе к содержательному тексту:

```
ТеТа Воронеж  Системная интеграция, автоматизация бизнес-процессов, ИТ-консалтинг
Системный администратор  Обслуживание и поддержка IT-инфраструктуры объектов
общепита. Работа с ПО iiko. Настройка, обслуживание ККТ и POS-терминалов.
```

### Шаг 2. Извлечение доменов из вакансии (`extract_domains_from_text` в `live_demo.py`)

Текст вакансии кодируется той же моделью, что и словарь. Из топ-кандидатов по сходству выбирается подмножество через **MMR**, чтобы домены не были все синонимами одной темы. Пример вывода (иллюстративно):

```
Домены запроса: ['тестирование программного обеспечения',
                 'автоматизация техподдержки',
                 'тестирование и обеспечение качества цифровых сервисов',
                 'автоматизация тестирования',
                 'sql-аналитика', ...]
```

Для вакансий дополнительно действует блоклист нерелевантных доменов (аналогично идее блоклиста для резюме).

### Шаг 3. Извлечение доменов из резюме (`extract_domains` в `parse_hh_resumes.py`)

Логика та же: эмбеддинг текста, MMR, пороги `min_sim` / `top_k`. Применяется `_DOMAIN_BLOCKLIST` для отсечения слишком универсальных меток, которые попадают почти в любое IT-резюме.

Итог по всем резюме сохраняется в JSON вида:

```json
{
  "961": ["тестирование программного обеспечения", "автоматизация тестирования", "..."],
  "962": ["разработка программного обеспечения", "..."]
}
```

### Шаг 4. Веса доменов вакансии (`estimate_domain_weights` в `live_demo.py`)

По секциям текста («обязательно», «желательно», нейтральные блоки) и локальной модели назначаются веса запросным доменам (обязательный / опциональный / нейтральный). Эти веса передаются в `scorer.rank(..., weights=...)`.

### Шаг 5. Скоринг (`scorer.rank`)

Краткая формула уровня идеи:

- для каждого запросного домена: `best_sim = max cosine(query_i, candidate_j)`;
- если `best_sim >= 0.72` - полноценный вклад плюс бонусы co-occurrence и таксономии;
- иначе - ослабленный вклад;
- `hit_rate` - доля запросных доменов с `best_sim >= 0.72`;
- итог усредняется с учётом весов и штрафа покрытия, затем нормализуется в [0, 1].

### Шаг 6. Вывод

В консоли или JSON: место кандидата, `score`, `hit_rate`, лучший матч по домену, шаблонное объяснение (`explain_text`) для топовых позиций.

Формат: обычный UTF-8 текст без разметки. Вакансию и резюме можно копировать из любого источника.

**Вакансия + готовый пул кандидатов** (по умолчанию `candidates_hh_full.json`, если есть, иначе fallback):

```bash
python score_file.py --vacancy /полный/путь/к/vacancy.txt
```

**Вакансия + конкретные резюме:**

```bash
python score_file.py --vacancy /полный/путь/к/vacancy.txt \
  --resume /полный/путь/к/cv_ivanov.txt /полный/путь/к/cv_petrova.txt
```

**Вакансия + все `.txt` в каталоге:**

```bash
python score_file.py --vacancy /полный/путь/к/vacancy.txt \
  --resume-dir /полный/путь/к/resumes/ --top 5
```

**JSON для интеграции:**

```bash
python score_file.py --vacancy /полный/путь/к/vacancy.txt \
  --resume-dir /полный/путь/к/resumes/ --json --out /полный/путь/к/result.json
```

## 10. Настройка LLM (`config.json`)

LLM используется **только** в скриптах подготовки данных (`build_expansions.py`, при необходимости таксономия), а не при обычном `rank()`.

Пример секции:

```json
{
  "llm": {
    "provider": "gemini",
    "api_key": "ВСТАВИТЬ_КЛЮЧ",
    "model": "gemini-2.5-flash",
    "base_url": ""
  }
}
```

Альтернативы: OpenAI-совместимый endpoint (Ollama, LM Studio, OpenRouter) - см. примеры в `_provider_examples` внутри `config.json`. Ключ можно передать переменной окружения `LLM_API_KEY` без хранения в файле.

**Не добавляйте `config.json` с секретами в публичный репозиторий.**

## 11. Использование как библиотеки

```python
from scorer import rank, explain, explain_text

results = rank(["финтех", "интеграция банковского ядра"], top=5)
for r in results:
    print(r["id"], r["score"], r["hit_rate"])

info = explain(6, ["финтех", "интеграция банковского ядра"])
print(explain_text(info))
```
